# 验收标准流程（权威）

> 本文件是「B站视频端到端验收」操作的**唯一权威标准**。
> 对应代码：`acceptance/flows/bilibili_video.py`；人读说明：`acceptance/README.md`。
> 三者（本文件 / 代码 / README）必须保持一致，**以本文件为准**。

## 范围

本流程**只负责操作编排**（拉起服务 / 喂输入 / 读输出 / 清理），**不做审核判定**。
审核（内容可信吗、优点是什么、能照搬哪几步、来源追到哪）由以后单独的流程负责，暂时不涉及。

## 标准流程（6 步）

1. **开关式防误删（单一开关）**：两个监控文件夹原本自动读文件并处理，现在改成「开关式」——用一个环境变量 `ACCEPTANCE_KEEP_FILES=1` 同时拉起炼真 / 熔知，两器同时受控（炼真保留中转①、熔知保留收件箱原文件），不移动、不删除中间文件。验收结束后 harness 停掉这两个服务即恢复常态（默认不带该变量 → 正常删除）。
2. **不干涉三项目**：只负责输入和输出，绝不手动干预馏析 / 炼真 / 熔知内部工作。三个项目各自的 watcher 必须自动进行。
3. **馏析出文件1**：给一个 B站网址 → 输入馏析 → 馏析输出第一个文件（中转①）。
4. **炼真 ×3（同输入）**：把馏析输出文件提交给炼真，等炼真产出文件后保存；重复 3 次 → 得到**同输入的 3 份炼真文件**（中转②）。
5. **熔知 ×3（同一份提交 3 次）**：先**随机选 1 份**（从 3 份炼真中 `random.choice`，选一次），把**同一份**提交给熔知，等熔知摄入数据库后，记录本次摄入的 **53 个字段**结果形成文件，**然后删除测试数据**。**重复 3 次**（共提交同一份 3 次）→ 得到 **3 份熔知字段文件**。
   - 关键点：「任意选择一个」= **选一次**，把选中的那一份**连续提交 3 次**（不是每轮独立抽，也不是「3 份各一次」）。
   - 「删除测试数据」= 每次记录字段后立即 `delete_doc`；删除会清掉该内容的 `content_hash`，故下一轮同内容可再次摄入（不会因熔知内容去重卡住）。
   - 前置约束（方案1）：炼真 ×3 的中转②在验收期被重定向到**暂存目录**（非熔知收件箱），故 step④ 阶段熔知**不会**自动摄入；只有 step⑤ 把选中的那份复制进收件箱才触发摄入。这既保留「三器自动」，又避免 step④/step⑤ 撞内容去重。
   - 前置约束：炼真必须先把 3 份全产完，熔知才能动（池未满不能抽）。
6. **共 7 份文件**：中转① ×1 + 炼真 ×3 + 熔知字段 ×3 = 7 份。
