> 研究日期：2026-07-04
> 目的：了解 AI 编程助手和 Agent 框架的最佳实践，找出对 SKILL.md 优化的具体启发

## 一、AI 编程助手的产品架构

### 1.1 核心产品对比

| 产品 | 规则文件机制 | 上下文管理策略 |
|------|-----------|----------------|
| **Claude Code** | `CLAUDE.md`（项目根目录）+ `.claude/skills/*.md`（按需加载） | `/clear` 清上下文、`/compact` 压缩、`--max-turns` 限制轮数 |
| **Cursor** | `.cursor/rules/*.mdc`（按范围拆分，旧版 `.cursorrules` 已废弃） | Agent 模式会用子代理隔离探索、@-mention 调用工作流 |
| **GitHub Copilot** | `copilot-instructions.md` | 工作区上下文自动管理 |
| **Windsurf** | `.windsurf/rules/*.md` | 类似 Cursor 的新格式 |

### 1.2 关键发现：规则文件的分层架构

**Claude Code 官方的 5 文件架构**（2026 年最佳实践）：

```
项目根目录/
├── CLAUDE.md       # Claude 专属行为规则（语气、格式、硬性禁止）
├── AGENTS.md       # 通用项目说明书（所有 AI 工具都能读）
├── MEMORY.md       # 跨会话持久记忆（决策日志、过往错误）
├── CONTEXT.md      # 当前会话状态（今日目标、当前分支、临时约束）
└── skills/          # 按需加载的技能模块
    ├── commit/SKILL.md
    ├── deploy/SKILL.md
    └── testing/SKILL.md
```

**各文件的职责分离**：

| 文件 | 加载时机 | Token 影响 | 内容特点 |
|------|----------|------------|----------|
| CLAUDE.md | 会话开始始终加载 | 中等（应 <100 行） | 行为规则、语气风格 |
| AGENTS.md | 会话开始始终加载 | 中等（应 <100 行） | 架构概览、构建步骤 |
| MEMORY.md | 始终或按需 | 随时间增长 | 追加式决策日志 |
| **skills/*.md** | **按需/动态加载** | **最小（仅需要时）** | 按任务拆分的深度指令 |
| CONTEXT.md | 会话范围 | 小（20-40 行） | 每次会话重写 |

> **核心启发 1**：SKILL.md 当前 463 行全部始终加载 → 应改为**按需加载的模块化结构**

### 1.3 防止「长对话遗忘流程」的策略

| 策略 | 具体做法 | 对 SKILL.md 的启发 |
|--------|-----------|-------------------|
| **两阶段工作流** | 计划阶段输出 `implementation-strategy.md`，执行阶段直接读文档 | SKILL.md 的「调试模式」应强制输出假设文档到文件，而非留在对话上下文 |
| **子代理隔离** | 冗长探索在独立上下文窗口运行，只返回摘要到主会话 | 复杂调试应考虑「探索子任务」分离 |
| **MCP 实时系统接入** | Playwright MCP（浏览器）、Postgres-MCP（数据库）让 Agent 有真实环境反馈，而非盲猜 | SKILL.md 应强调「用真实环境验证假设」，而非纯代码推理 |
| **Hooks 确定性防护** | 用 shell 脚本做确定性检查，无法被提示词绕过 | 关键规则（如「禁止改 schema」）应用 Hooks 而非提示词 |

### 1.4 Context 管理的五个核心原则

1. **精准锚定**：告诉 Agent 精确文件路径和行号，而非让它搜索（降低 context 消耗 10-50 倍）
2. **分块处理**：大任务拆成独立 context 的子任务，交付物写文件
3. **主动压缩**：context 消耗 70% 时主动 compact，而非等截断
4. **信息密度分级**：Tier 1（完整保留）/ Tier 2（压缩保留）/ Tier 3（完全丢弃）
5. **会话边界设计**：不同任务用不同 session，避免无关历史污染

> **核心启发 2**：SKILL.md 当前要求「每次全读」→ 应改为「分级加载 + 按需 grep」

## 二、AI Agent 编程框架

### 2.1 主流框架对比

| 框架 | Agent Loop 设计 | 记忆系统 | 适用场景 |
|--------|---------------|----------|----------|
| **LangChain/LangGraph** | 有向循环图，状态机控制 | 短期：context window；长期：向量库 RAG | 复杂工作流、需人工审核节点 |
| **AutoGen** | 多 Agent 协作，角色分工 | 各 Agent 独立上下文 | 软件工程、内容流水线 |
| **OpenHands** | 事件流 + Condenser 压缩 | 三层：View 过滤 → Condenser 压缩 → ConversationMemory 格式化 | 自主编程任务 |
| **CrewAI** | 角色驱动，顺序/层次流程 | 各 Agent 独立记忆 | 业务流程自动化 |

### 2.2 OpenHands 的记忆系统架构（最相关）

OpenHands 采用**三层记忆模型**解决「有限上下文窗口 vs 完整历史信息」矛盾：

```
Agent.step()
  → Condenser.condensed_history()   // 压缩历史（有损）
    → View.from_events()            // 过滤噪声事件（NullAction 等）
      → ConversationMemory.process_events()  // 转换为消息格式
        → LLM 推理
```

**Condenser 的压缩策略**：
- `ConversationWindowCondenser`：固定窗口，只保留最近事件
- `LLMSummarizingCondenser`：用 LLM 对早期历史做摘要

> **核心启发 3**：SKILL.md 的「假设追踪表」本质上是压缩调试历史的机制——应明确要求「调试历史写到文件，不在对话里累积」

### 2.3 「证据链」和「假设追踪」在框架中的对应机制

| 框架 | 对应机制 | 实现方式 |
|--------|----------|----------|
| **LangGraph** | 状态机中的 `reflection` 节点 | 每次执行后强制进入反思节点，对比预期 vs 实际 |
| **OpenHands** | `ConversationMemory` 维护工具调用与响应的关联 | 通过 `tool_call_id` 关联，确保上下文完整性 |
| **AutoGen** | `GroupChat` 中的 `ConversableAgent` 反思 | Agent 之间互相审查输出 |

> **核心启发 4**：SKILL.md 的「证据链」是好的，但缺少**强制反思节点**——应要求「每次假设验证后，显式输出反思结论」

## 四、对 SKILL.md 优化的具体启发

### 4.1 文件结构重构（高优先级）

**目标**：从 463 行单文件 → 分层模块化结构

```
workflow/skills/project-management-workflow/
├── SKILL.md                # 核心规则摘要（<100 行，始终加载）
├── SKILL-CHEATSHEET.md   # 场景 SOP、Gate 格式、审查清单（按需 grep）
├── debug-protocol.md       # 调试协议全文（调试模式下按需加载）
├── frontend-rules.md      # 前端验证规则（前端任务时按需加载）
└── context-management.md  # 上下文管理策略（长任务时按需加载）
```

**加载策略**：

| 场景 | 加载的文件 | 估计 Token |
|--------|-----------|-------------|
| Bug 修复 | SKILL.md（核心规则）+ 今天日志 | ~2000 token |
| 调试模式 | SKILL.md + debug-protocol.md | ~3000 token |
| 前端任务 | SKILL.md + frontend-rules.md | ~2500 token |
| 功能开发 | SKILL.md + SKILL-CHEATSHEET.md（grep 相关段落） | ~2000 token |

### 4.2 内容优化（高优先级）

**从竞品学到的具体改法**：

| 当前问题 | 竞品做法 | SKILL.md 改法 |
|----------|----------|----------------|
| 463 行始终加载 | Claude Code: <100 行 + 按需 `@` 引用 | 拆分成 5 个文件，主文件只留绝对规则摘要 + 路径索引 |
| 调试历史在对话里累积 | OpenHands: Condenser 压缩到文件 | 调试模式强制「假设追踪表写到文件」，不在对话里保留 |
| 没有强制反思节点 | LangGraph: reflection 节点 | 证据链结论后加「反思检查点」：这个结论是否改变我的方案？ |
| 上下文管理缺失 | Claude Code: `/compact` + `--max-turns` | 新增「上下文管理规则」段落：>15 轮建议 compact 或开新对话 |
| 无法验证假设真实性 | Claude Code: MCP 实时系统 | 调试协议强调「用真实环境验证」（浏览器/MCP/测试）而非纯代码推理 |

### 4.3 「证据链」机制的增强建议

**当前 SKILL.md v5.2 的证据链**：

```
## 证据链
现象：[用户报告的症状]
假设：[我认为根因是 X]
验证方法：[我怎么证明假设对不对]
验证结果：[证据支持/推翻了假设]
结论：[根因确认 = X / 假设排除，换下一个]
```

**从竞品增强后的版本**（参考 OpenHands 的 View 过滤 + LangGraph 的 reflection）：

```
## 证据链（必须写到文件 `debug-log.md`）

### 现象
[用户报告的症状 — 客观描述，不带假设]

### 假设与验证
| # | 假设 | 验证方法 | 验证结果 | 状态 | 反思结论 |
|---|------|---------|---------|------|----------|
| 1 | [假设 A] | [方法] | [结果] | ❌ 排除 | [这次验证教会了我什么] |
| 2 | [假设 B] | [方法] | [结果] | ... | ... |

### 当前根因（第 3 行后必须停下来确认）
[根因 = X，置信度：高/中/低]
[如果置信度 ≠ 高，必须列出还能验证什么]

### 下一步行动
- [ ] 改代码：[具体改哪里、预期效果]
- [ ] 验证：[具体怎么证明改好了]
```

### 4.4 竞品功能对照表（SKILL.md 缺失的能力）

| 能力 | Claude Code | Cursor | OpenHands | SKILL.md 当前 | 建议 |
|------|------------|--------|------------|--------------|------|

| 上下文压缩 | ✅ `/compact` | ✅ Agent 模式自动 | ✅ Condenser | ❌ 无 | 新增压缩规则 |

| MCP 接入验证 | ✅ Hooks + MCP | ✅ MCP 支持 | ✅ 工具调用 | ❌ 无（纯代码推理） | 新增「真实环境验证」要求 |

## 六、一句话总结

**竞品怎么做**：规则文件 <100 行始终加载 + 按需加载的技能模块 + 上下文压缩 + 真实环境验证（MCP）+ 确定性 Hooks 防护。

**SKILL.md 当前差距**：463 行单文件全量加载、无上下文压缩机制、调试靠纯代码推理而非真实环境验证、关键规则只靠提示词（可被绕过）。

**最急的改法**：把 SKILL.md 拆成「主文件（<100 行）+ 3 个按需加载的模块文件」，参考 Claude Code 的 5 文件架构。
