# Workflow · 项目管理流程 — 蓝图

> 这是项目管理流程的宪法。只写方向和原则，不写技术细节。
> 具体怎么做，看 FLOWCHART.md 和 SKILL.md。

---

## 愿景

让每个编程任务都有迹可循、有理可依、有错可改。

**核心理念**：用户给指令 → AI 探索 → 蓝图对齐 → 拆解任务 → 讨论方案 → MVP 审查 → 执行 → 自审 → 验证 → 用户验收 → git commit + push → 蓝图反哺 → 文档更新 → 归档。

这是一条完整闭环。每个环节都有明确的进出标准，绝不跳过。

## 核心原则

1. **蓝图先行** — 每个项目必须有 BLUEPRINT.md（宪法）和 FLOWCHART.md（数据流）。先定 what/why，再定 how。没有蓝图的代码，就是没有铅垂线的大楼。
2. **节点映射** — 每个实现任务必须在 FLOWCHART.md 的数据流中找到对应节点（📍N1 格式）。找不到节点 → 红牌警告。
3. **MVP 优先** — 设计阶段主动跳过边缘 case，执行阶段防偏检查，卡住就问「跳过它对 MVP 有影响吗？」
4. **反馈闭环** — 所有失败路径都有回路（验证回路、自审回路、阻塞回路、调试刹车回路、蓝图回路），所有跳过的任务记入 PROJECT_PLAN.md 已知问题段落。
5. **调试纪律** — 调试时 MUST 先输出证据链再动手，假设追踪表第 3 行 MUST 停下来询问用户。

## 当前重心

流程 v5.0 已落地（U 型注意力优化 + 调试协议 + 分级加载 + 三条路径）。当前重心：**推广使用** — 确保所有 OpusMagnum 子项目（Citrinitas / Nigredo / Albedo / Rubedo）统一使用此流程。

## 边界

- 不覆盖非编程任务（闲聊/调研走短路径，不创建任务）
- 不强制工具选择（不规定用哪个编辑器/测试框架）
- 只支持单会话任务队列（多会话并行在远期）
- 手机端交互不在范围

## 验收标准

- 任何编程任务都能按 Phase 链走通（Pre-flight → Phase 5）
- 调试模式能阻止"一个问题打转"（假设表第 3 行刹车）
- 每个任务可追溯至 FLOWCHART 节点
- 每次变更都有 CHANGELOG + 蓝图反哺
- 技术债有 PROJECT_PLAN.md 追踪，不丢失

---

## 管理文件体系

本流程要求每个项目维护以下文件：

| 文件 | 职责 |
|------|------|
| `BLUEPRINT.md` | 项目宪法 — 愿景/核心原则/当前重心/边界/验收标准 |
| `FLOWCHART.md` | 流程框图 — Mermaid 主干图 + 节点输入输出表 + 任务映射 |
| `CHANGELOG.md` | 版本变更记录（Keep a Changelog） |
| `PROJECT_PLAN.md` | 功能路线图 + 设计决策 + 已知问题（含技术债） |

> 技术债直接记录在 PROJECT_PLAN.md 的「已知问题」段落中，NEVER 创建独立的 TECH_DEBT.md 文件。

---

> *Build in phases. Align to blueprint. Ship relentlessly.*
