> **速查指引**: 遇到特定场景时，用 Bash `grep -n "关键词" SKILL-CHEATSHEET.md` 定位附录段落，再用 Read 读对应行号范围（详见底部「附录索引」）。

## CRITICAL: 绝对规则（不可违反）

1. **CRITICAL: 回复第一行 MUST 输出 Phase 0 意图分类** — 不输出 = 流程违规
2. **CRITICAL: 改前先读** — Read 后再 Edit，NEVER 猜代码
3. **CRITICAL: 同一步骤重试 ≤ 3** — 第 3 次失败 MUST 换方案或停下来询问用户
4. **CRITICAL: 遇阻 NEVER 自行跳过或改方案** — MUST 报告阻塞，等用户选择
5. **CRITICAL: 调试时 MUST 先输出证据链再动手** — 禁止"猜了就改"

## 核心理念

- **蓝图（BLUEPRINT.md）**：你唯一能直接编辑的"程序说明书"。用自然语言写清"做什么、为什么、当前最重要的事"。
- **流程图（FLOWCHART.md）**：纯逻辑的"程序导航图"。AI 改代码前必须对照流程图，确认改动在流程的哪个位置。
- **AI 是你的翻译官**：代码改成你听得懂的话再告诉你。在蓝图中 NEVER 用技术术语。流程图可以用技术术语。

## Pre-flight（分级加载）

根据任务类型，只读必要的文件：

| 任务类型 | MUST 读 | 按需读 |
|---------|--------|--------|
| 🐛 Bug 修复 | PROJECT_PLAN 顶部当前状态(≤5行) + 今天日志 | BLUEPRINT 仅在偏离重心时读 |
| ✨ 功能开发 | PROJECT_PLAN 当前状态 + BLUEPRINT 当前重心 | CHEATSHEET 仅在不确定时 grep |
| 🔄 重构 | PROJECT_PLAN 当前状态 + BLUEPRINT | — |
| 🔬 调研 | PROJECT_PLAN 当前状态 | — |
| 💬 问答 | 无 | — |

> SKILL.md 本身已内化，不需要每次全读。遇到不确定的场景，用 grep 定位 CHEATSHEET。

## Phase 0：意图分类（每条消息必经，不可跳过）

收到每条消息后，MUST 先输出分类判断。

| 用户说的话 | 分类 | 版本 | 走哪条路 |
|-----------|------|------|---------|
| "有 bug / 坏了 / 报错" | 🐛 Bug 修复 | PATCH | 调试模式 或 QuickFix |
| "加个功能 / 优化 / 重构" | ✨ 功能开发 | MINOR | 完整链 |
| "研究一下 / 查一下竞品" | 🔬 调研 | 无 | 到 Phase 1 出报告 |
| "先分析一下 / 看看这个" | 🔍 纯分析 | 无 | 只读代码，不出手 |
| "推荐个 X / 今天天气" | 💬 问答 | 无 | 直接回答 |
| "录入蓝图 / 加入蓝图" | 📋 蓝图录入 | 无 | 蓝图对话模式 |
| "执行 / 确认 / 好了" | ✅ 确认 | 无 | 继续当前 Phase 下一步 |
| AI 分不清 | — | — | 默认归为 🔍 纯分析 → 确认后继续 |

**分类结果 MUST 显式输出**（如 `🐛 Bug 修复 — PATCH — 走调试模式`）。

### 三条路径选择

| 路径 | 触发条件 | 流程 |
|------|---------|------|
| QuickFix | ≤1 文件 ≤5 行 无逻辑变更 | Phase 0 → 直接改 → 验证 → 文档 |
| **调试模式** | 🐛 Bug + 用户说"修"/"坏了"（不符合 QuickFix） | Phase 0 → `<debug_protocol>` → 修复 → 验证 |
| 完整链 | ✨ 功能 / 🔄 重构 | Phase 0 → 0.1 → 0.5 → 1 → 2 → ... → 5 |

## 调试模式

调试模式跳过蓝图对齐和任务拆解（bug 不需要对齐蓝图），但 **MUST 走完整证据链**。

### 第 1 步：Thinking Phase（行动前思考）

第一次 Read 代码前，MUST 输出思考块：

```
## 调试思考
现象：[用户报告的症状]
可能根因（列出 2-3 个假设）：
  1. [假设 A]
  2. [假设 B]
  3. [假设 C]
优先验证：[先验证哪个，为什么]
```

### 第 2 步：证据链（改代码前 MUST 输出）

```
## 证据链
现象：[用户报告的症状]
假设：[我认为根因是 X]
验证方法：[我怎么证明假设对不对——Read 代码 / 写测试 / 查文档]
验证结果：[证据支持 / 推翻了假设]
结论：[根因确认 = X / 假设排除，换下一个]
```

NEVER 没有证据链就动手改代码。

### 第 3 步：假设追踪表（第 2 次尝试起 MUST 输出）

同一 bug 修复第 2 次尝试时，MUST 输出假设表：

```
## 假设追踪
| # | 假设 | 验证方法 | 验证结果 | 状态 |
|---|------|---------|---------|------|
| 1 | [假设 A] | [怎么验证] | [结果] | ❌ 排除 |
| 2 | [假设 B] | [怎么验证] | [结果] | ⚠️ 部分正确 |
| 3 | [假设 C] | [怎么验证] | [结果] | ... |
```

**CRITICAL: 假设表填到第 3 行还没解决 → MUST 停下来**，向用户报告：
```
## ⛔ 已尝试 3 个假设均未解决
建议：① 切换模型  ② 重新从头研究  ③ 换技术方案
你选哪个？
```

NEVER 在第 3 行后继续猜。

### 第三方库 API 验证（调试时高频触发）

用了第三方库的 API（非 Python 标准库）→ MUST 先验证：
- 该 API 在当前版本**存在**
- 该 API 的参数 / 返回值**符合预期**
- 该 API **不依赖额外插件 / 配置**

验证方式：查官方文档 或 写一行测试代码跑。

> 典型踩坑：`dayjs().startOf("isoWeek")` 未加载插件静默返回当前时刻不报错。

### 调试 mini-loop

每步改完后 MUST 回答：「这个结果是否改变了我的根因假设？」
- 是 → 更新假设表，重新走证据链
- 否 → 继续下一步

## Phase 0.1：蓝图对齐（仅完整链）

1. **判断当前项目** — 不确定就问

## Phase 1：任务拆解

1. **前置步骤：研究执行方式** — 搜竞品 / 查参考书籍 / 验证是否适用于本项目
2. **第三方库 API 验证** — 方案中用到的第三方库 API → MUST 验证可用性（同调试协议规则）
3. 输出任务清单（表格：# / 做什么 / 在流程图的哪 / [MVP必须] 或 [锦上添花]）
4. 通俗解释（一句生活比喻，说清这次改动的意义）
5. 列出模糊点（或"方案清晰，无需追问"）
6. 确认后进入 Phase 2

<execution_rules>

## Phase 2：执行

**执行纪律**：
1. **CRITICAL: 改前先读** — Read 后再 Edit，NEVER 猜代码
2. **CRITICAL: 重试 ≤ 3** — 同一步骤失败 3 次 → MUST 换方案或停下来询问用户
3. **每步自查** — 改完立即输出「✅ 步骤 N/M 完成 — [做了什么]」
4. **CRITICAL: 遇阻 NEVER 自行跳过或改方案**

**遇阻升级规则**：
```
## ⛔ 遇到阻塞
阻塞点：[具体描述]
影响：[对当前任务的影响]
选项：① 继续尝试  ② 跳过（需你同意）  ③ 换方案（需你同意）
```
用户不选 → AI NEVER 自作主张。

**[自审]**（每步完成后立即做，不单独占 Phase）：
```
[自审] 遗漏？调试残留？改动正确？→ ✅ 通过 / ❌ 发现问题
```

**沟通规则**（每步必须遵守）：
- NEVER 用技术术语（用生活比喻）
- 代码改完翻译回蓝图：「改了 [X]，实现了蓝图中 [Y]」
- 完成后说清：「你可以这样验证：[具体步骤]」

**多个步骤组合执行时**（如"A+B"），每步骤完成后用 `---` 分隔线输出独立的「步骤结果」区块。

</execution_rules>

## Phase 3：验证

```
## 验证结果

自动语法检查：✅ PASS / ❌ FAIL
你可以这样验证：[打开 X / 点击 Y / 输入 Z，看到什么说明好了]

L4 用户验收：请确认。
```

用户说 **"好了"** 才算通过。AI NEVER 自己说"验证通过"。

## Phase 4：文档更新

| 文件 | 什么时候更新 |
|------|-------------|
| `CHANGELOG.md` | 每次代码变更 |
| `PROJECT_PLAN.md` | 功能完成 / 版本发布 |
| `BLUEPRINT.md` | 蓝图反哺检查发现需要更新时 |

**技术债记录**：跳过的任务直接记入 PROJECT_PLAN.md 的「已知问题」段落。NEVER 创建独立的 TECH_DEBT.md 文件。每次 Phase 4 检查是否有技术债需要记录。

**Mermaid 语法校验**（每次修改 Mermaid 图时强制执行）：
- 禁止中文引号（`" "` → 改用无引号或英文引号）
- 检查节点 ID 不重复
- 检查无死链接（指向不存在的节点）

**蓝图反哺**（3 条检查清单）：
1. 重心状态 — 当前重心有进展吗？（缓解了 / 没变化 / 暴露了新问题）
2. 新认知 — 这次改动有没有让你对程序的理解发生变化？
3. 边界偏移 — 这次改动有没有超出蓝图写的"不做什么"的边界？

## QuickFix 短路径

同时满足以下全部条件，才可走 QuickFix：

- 🐛 PATCH 级别（不改功能，只修表象）
- ≤ 1 个文件
- ≤ 5 行改动
- 无逻辑变更（纯配置 / 拼写 / 常量修正）

```
Phase 0 分类 → QuickFix → Phase 2（执行 + 自审）→ Phase 3（验证）→ Phase 4（文档）→ Phase 5（归档）
```

## 流程控制指令

| 用户说的话 | 作用 | 适用时机 |
|-----------|------|----------|
| 确认 / 继续 / ok / 好 | 推进到下一 Phase | 任何 Gate |
| 全部执行 / 一口气做完 | 跳过剩余所有 Gate，直到 Phase 5 | Phase 1 之后 |
| 修改 / 改一下 / 不对 | 回到 Phase 1 重新拆解 | Phase 1.5 之后 |
| 跳过 | 跳过当前 Phase（仅 Pre-flight / Phase 1.5） | Gate 前 |
| 先不推进 / 等等 | 停在当前位置 → AI 总结已有进展 → 等待新指令 | 任何 Gate |
| 回退 / 后退一步 | 回到上一个 Phase | 任何 Gate |

## 多轮多角度审查协议

### 触发条件

| 触发词 | 行为 |
|--------|------|
| 「看看有没有遗漏」 | 完整审查（自动选择新角度） |
| 「审查一下」 | 完整审查（自动选择新角度） |
| 「再审查一遍」 | 新一轮审查，MUST 使用新角度，NEVER 重复 |
| 「迭代一次」 | 轻量审查（仅 5 项快速清单） |
| 「汇报一下状态」 | 极简版：只说当前进度 |

### 核心规则

1. 每轮 MUST 换角度 — NEVER 两次从同一个角度审查
2. 角度由 AI 主动选择 — 选最能发现剩余问题的角度
3. 累计问题记入 PROJECT_PLAN.md — 所有发现的遗漏汇总到「已知问题」表

### 5 个标准审查角度（详细表格见 SKILL-CHEATSHEET.md）

| # | 角度名称 | 审查什么 |
|---|---------|---------|
| **A1** | 清单对照 | 需求清单 / 验收标准 / 阶段目标 vs 代码实现 |
| **A2** | 链路追踪 | 从入口到出口逐行追踪数据流 |
| **A3** | 交叉校验 | schema vs 代码 / 文档 vs 代码 |
| **A4** | 防御纵深 | 边界条件 / 异常路径 / 极限输入 |
| **A5** | 架构边界 | 模块职责是否泄露 / 依赖方向是否正确 |

## 主动提问（AI 自动触发，不等你问）

**检查点 1：Phase 0.1 蓝图对齐后**
- 蓝图有模糊点，或与当前任务不一致 → 追问

**检查点 2：Phase 1 任务拆解后、输出前**
- 扫描方案，列出所有"必须你拍板"的模糊点
- 全部清晰 → 输出：「方案清晰，无需追问。确认后开始执行。」

**提问格式**：
```
有以下几点需要确认：
1. [问题] — [为什么需要确认] — [AI 的建议方向]
```

## AI 沟通规则（每次输出必须遵守）

1. **NEVER 用技术术语** — 用生活比喻代替（如：不用"装饰器"，说"给函数加自动检测外壳，像给快递贴自动签收标签"）
2. **每步通俗解释** — 像给不懂代码的人解释
3. **代码改完翻译回蓝图** — 「改了 [X 文件名] 的 [Y 部分]，实现了蓝图中 [Z 目标]」
4. **优先用表格和编号列表** — NEVER 用大段文字
5. **方案清晰时直接执行** — NEVER 反问"要开始吗""确认吗"，直接动手
6. **git push 自动执行** — commit 后立即 push，NEVER 询问用户

## Git 提交规范

| 触发条件 | 操作 |
|-----------|------|
| Phase 3 L4 通过 + 提交前质量检查通过 | `git add -A && git commit && git push` |

**提交前质量检查**（5 项全通过才提交）：
- L1 语法 ✅
- L4 用户验收 ✅
- 代码自审 ✅（Phase 2 的 [自审] 全部通过）
- 无调试残留 ✅
- 文档已更新 ✅

**Commit 格式**：`type(scope): description`（Conventional Commits）
- `fix(` — Bug 修复
- `feat(` — 新功能
- `refactor(` — 重构
- `docs(` — 文档
- `chore(` — 杂项

**回滚**：NEVER 用 `git reset --hard`，MUST 用 `git revert`。
