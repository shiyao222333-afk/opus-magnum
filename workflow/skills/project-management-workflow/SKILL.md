# 项目管理 Workflow v4.0

> ⚠️ MANDATORY — 所有开发对话必须首先读取此文件。

> **速查指引**: 遇到特定场景时，用 Bash `grep -n "关键词" SKILL-CHEATSHEET.md` 定位附录段落，再用 Read 工具读对应行号范围（详见本文档底部「附录索引」）。

> **Pre-flight 简化**: 先读 `PROJECT_PLAN.md` 顶部 `## 当前状态`（≤5行），再读 `BLUEPRINT.md`。

---

## 核心理念

- **蓝图（BLUEPRINT.md）**：你唯一能直接编辑的"程序说明书"。用自然语言写清"做什么、为什么、当前最重要的事"。
- **流程图（FLOWCHART.md）**：纯逻辑的"程序导航图"。AI 改代码前必须对照流程图，确认改动在流程的哪个位置。
- **AI 是你的翻译官**：代码改成你听得懂的话再告诉你。**在蓝图中，禁止用技术术语**。流程图可以用技术术语。

---

## Pre-flight（每次任务开始前）

1. 读 `PROJECT_PLAN.md` 顶部 `## 当前状态`（≤5行，快速了解当前版本和下一个版本）
2. 读项目的 `BLUEPRINT.md`（不存在则询问是否创建）
3. 读 `~/.workbuddy/MEMORY.md`（全局规则）
4. 读今天的 memory 日志
   - 如果最后一条含 **"待实施"** → 主动问：「要继续昨天的 [X] 吗？」

---

## 中断恢复（对话中途纠正/追问/补充时触发）

收到纠正/追问 → 回到 Pre-flight（读蓝图+流程图+日志）→ 重新 Phase 0 分类 → 输出新分类 → 继续

---

## Phase 0：意图分类（每条消息必经，不可跳过）

收到你的每条消息后，**必须先输出分类判断**，才能继续做其他事。

| 你说的话 | 分类 | 版本影响 | 走哪条链 |
|-----------|--------|----------|-----------|
| "有 bug / 坏了 / 报错" | 🐛 Bug 修复 | PATCH | 完整链 |
| "加个功能 / 优化 / 重构" | ✨ 功能开发 | MINOR | 完整链 |
| "研究一下 / 查一下竞品" | 🔬 调查/研究 | 无版本 | 到 Phase 1 出报告 |
| "先分析一下 / 看看这个" | 🔍 纯分析 | 无版本 | 只读代码，不出手 |
| "推荐个 X / 今天天气" | 💬 纯问答 | 无版本 | 直接回答 |
| "录入蓝图 / 加入蓝图" | 📋 蓝图录入 | 无版本 | 蓝图对话模式 |
| "执行 / 确认 / 好了" | ✅ 确认 | 无版本 | 继续当前 Phase 的下一步 |
| AI 分不清 | — | — | 默认归为 🔍 纯分析 → 确认后继续 |

**分类结果必须显式输出**（如 `🔬 研究 — 到 Phase 1 出报告`）。

不确定分类 → 用 Bash 执行：`grep -n "场景" SKILL-CHEATSHEET.md` 定位详细 SOP。

---

## Phase 0.1：蓝图对齐（必须）

1. **判断当前项目** — 根据上下文推断。不确定就问。
2. 读该项目 `BLUEPRINT.md`，输出：「蓝图中「当前重心」是 [X]。这次任务 [与重心一致 / 偏离重心（必要维护）/ 偏离重心（需讨论）]。」
3. **跨项目检查**（一句话）：改动涉及 `.env` / `config/` / `requirements.txt` → 「⚠️ 可能影响其他项目，需要确认吗？」

---

## Phase 0.5：代码探索

改前必须先 Read/Grep/Glob 代码，理解项目结构和现有实现。未探索就动手 → 流程违规。

---

## Phase 1：任务拆解

1. **前置步骤：研究执行方式** — 给出方案前，必须先研究具体怎么做（搜竞品 / 查参考书籍 / 验证是否适用于本项目）。
2. 输出任务清单（极简表格：# / 做什么 / 在流程图的哪）。
3. 通俗解释（一句生活比喻，说清这次改动的意义）。
4. 主动列出模糊点（或"方案清晰，无需追问"）。
5. 确认后进入 Phase 2。

---

## Phase 2：执行

**执行纪律**：
- 改前先读（Read 后再 Edit，不猜代码）
- 重试 ≤ 3（同一步骤失败 3 次 → 必须换方案，禁止同一思路反复试）
- 每步自查（改完立即输出「✅ 步骤 N/M 完成 — [做了什么]」）

**[自审]**（每步完成后立即做，不单独占 Phase）：
```
[自审] 遗漏？调试残留？改动正确？→ ✅ 通过 / ❌ 发现问题
```

**沟通规则**（每步必须遵守）：
- 禁止技术术语（用生活比喻）
- 代码改完翻译回蓝图：「改了 [X]，实现了蓝图中 [Y]」
- 完成后说清：「你可以这样验证：[具体步骤]」

---

## Phase 3：验证

```
## 验证结果

自动语法检查：✅ PASS / ❌ FAIL
你可以这样验证：[打开 X / 点击 Y / 输入 Z，看到什么说明好了]

L4 用户验收：请确认。
```

你说 **"好了"** 才算通过。AI 不能自己说"验证通过"。

---

## Phase 4：文档更新

**更新文件**（极简，只更新必要的）：

| 文件 | 什么时候更新 |
|------|-------------|
| `CHANGELOG.md` | 每次代码变更 |
| `PROJECT_PLAN.md` | 功能完成 / 版本发布 |
| `BLUEPRINT.md` | 蓝图反哺检查发现需要更新时 |

**Mermaid 语法校验**（每次生成/修改 Mermaid 图时强制执行）：
- 禁止中文引号（`" `" → 改用无引号或英文引号）
- 检查节点 ID 不重复
- 检查无死链接（指向不存在的节点）

**蓝图反哺**（3 条检查清单）：
1. 重心状态 — 当前重心有进展吗？（缓解了 / 没变化 / 暴露了新问题）
2. 新认知 — 这次改动有没有让你对程序的理解发生变化？
3. 边界偏移 — 这次改动有没有超出蓝图写的"不做什么"的边界？

---

## Phase 5：记忆归档

写 **一句**到今天的 memory 日志：

```
- [做了什么] — 结果：[成功/阻塞/待实施]
```

如果还有后续任务 → 末尾加 **"待实施：[下一步]"**，方便明天恢复。

---

## QuickFix 短路径

满足以下全部条件时，走快速通道（跳过 Phase 0.1 / Phase 1）：

- 🐛 PATCH 级别（不改功能，只修表象）
- ≤ 1 个文件
- ≤ 5 行改动
- 无逻辑变更（纯配置 / 拼写 / 常量修正）

```
Phase 0 分类 → QuickFix → Phase 2（执行 + 自审）→ Phase 3（验证）→ Phase 4（文档）→ Phase 5（归档）
```

## 流程控制指令

| 你说的话 | 作用 | 适用时机 |
|-----------|------|----------|
| 确认 / 继续 / ok / 好 | 推进到下一 Phase | 任何 Gate |
| 全部执行 / 一口气做完 | 跳过剩余所有 Gate，直到 Phase 5 | Phase 1 之后 |
| 修改 / 改一下 / 不对 | 回到 Phase 1 重新拆解 | Phase 1.5 之后 |
| 跳过 | 跳过当前 Phase（仅允许 Pre-flight / Phase 1.5） | Gate 前 |
| 先不推进 / 等等 | 停在当前位置 → AI 总结已有进展 → 等待你的新指令 | 任何 Gate |
| 回退 / 后退一步 | 回到上一个 Phase | 任何 Gate |

## 多轮多角度审查协议

### 触发条件

你说以下任意一句时启动审查：

| 触发词 | 行为 |
|--------|------|
| 「看看有没有遗漏」 | 完整审查（自动选择新角度） |
| 「审查一下」 | 完整审查（自动选择新角度） |
| 「再审查一遍」 | 新一轮审查，**必须使用新角度，禁止重复** |
| 「迭代一次」 | 轻量审查（仅 5 项快速清单） |
| 「汇报一下状态」 | 极简版：只说当前进度 |

### 核心规则

1. **每轮必须换角度** — 不能两次从同一个角度审查
2. **角度由 AI 主动选择** — 选最能发现剩余问题的角度
3. **累计问题记入 PROJECT_PLAN.md** — 所有发现的遗漏汇总到「已知问题」表

### 5 个标准审查角度（详细表格见 SKILL-CHEATSHEET.md）

| # | 角度名称 | 审查什么 |
|---|---------|---------|
| **A1** | 清单对照 | 需求清单 / 验收标准 / 阶段目标 vs 代码实现 |
| **A2** | 链路追踪 | 从入口到出口逐行追踪数据流 |
| **A3** | 交叉校验 | schema vs 代码 / 文档 vs 代码 |
| **A4** | 防御纵深 | 边界条件 / 异常路径 / 极限输入 |
| **A5** | 架构边界 | 模块职责是否泄露 / 依赖方向是否正确 |

## 主动提问（AI 自动触发，不等你问）

在以下两个检查点，AI **必须**主动列出需要你确认的问题：

**检查点 1：Phase 0.1 蓝图对齐后**
- 如果蓝图有模糊点，或与当前任务不一致 → 追问

**检查点 2：Phase 1 任务拆解后、输出前**
- 扫描方案，列出所有"必须你拍板"的模糊点
- 如果确实全部清晰 → 输出：「方案清晰，无需追问。确认后开始执行。」

**提问格式**：
```
有以下几点需要确认：
1. [问题] — [为什么需要确认] — [AI 的建议方向]
```

---

## AI 沟通规则（每次输出必须遵守）

1. **禁止技术术语** — 用生活比喻代替（如：不用"装饰器"，说"给函数加自动检测外壳，像给快递贴自动签收标签"）
2. **每步通俗解释** — 像给不懂代码的人解释
3. **代码改完翻译回蓝图** — 「改了 [X 文件名] 的 [Y 部分]，实现了蓝图中 [Z 目标]」
4. **优先用表格和编号列表** — 不用大段文字
5. **"给我看看"根据上下文判断** — 改代码的任务 → 看效果（启动服务，给访问地址）；规划任务 → 看蓝图（输出蓝图相关段落）
6. **方案清晰时直接执行** — 不反问"要开始吗""确认吗"，直接动手

## Git 提交规范

| 触发条件 | 操作 |
|-----------|------|
| Phase 3 L4 通过 + 提交前质量检查通过 | `git add -A && git commit` |

**提交前质量检查**（5 项）：
- L1 语法 ✅
- L4 用户验收 ✅
- 代码自审 ✅（Phase 2 的 [自审] 全部通过）
- 无调试残留 ✅
- 文档已更新 ✅

**Commit 格式**：`type(scope): description`（Conventional Commits）
- `fix(` — Bug 修复
- `feat(` — 新功能
- `refactor(` — 重构
- `docs(` — 文档

**回滚**：永远用 `git revert`（不用 `git reset --hard`）。

| 改动类型 | 版本号变化 | 示例 |
|-----------|-----------|---------|
