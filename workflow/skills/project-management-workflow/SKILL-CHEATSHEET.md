## 场景 SOP 速查

### 场景 1：Bug 修复
- **触发词**：bug、修、坏、不行、报错、异常、没反应
- **路由**：🐛 PATCH → 完整链
- **特殊规则**：
  - Phase 0.1 标注为「偏离重心（必要维护）」
  - Phase 3 L4 必须你复现确认
  - 支持 QuickFix（≤1 文件 ≤5 行）

### 场景 2：新功能
- **触发词**：加个、做个、实现、支持、优化
- **路由**：✨ MINOR → 完整链
- **特殊规则**：
  - Phase 1 区分 [MVP必须] / [锦上添花]
  - MVP 审查：这个方案是否满足 MVP 原则？

### 场景 3：调查/研究
- **触发词**：研究一下、查一下、看看竞品、调研
- **路由**：🔬 调查/研究 → 到 Phase 1 出报告
- **SOP 详细步骤**：
  1. 搜索 GitHub 同类项目（先搜，不能凭自己知识回答）
  2. 查阅相关参考书籍（如有）
  3. 输出分析报告（对比表 + 建议）
  4. 明确标注「等待你确认是否实施」

### 场景 4：纯分析
- **触发词**：先分析一下、看看这个、给我讲讲、怎么实现的
- **路由**：🔍 纯分析 → 只读代码，不出手
- **退出条件**：你明确说「开始做」才进入 Phase 2

### 场景 5：纯问答
- **触发词**：推荐、今天、怎么看、什么是
- **路由**：💬 纯问答 → 直接回答，不走链

### 场景 6：蓝图录入
- **触发词**：录入蓝图、加入蓝图、更新蓝图、蓝图改成
- **路由**：📋 蓝图录入 → 蓝图对话模式
- **三步**：语义分析 → 展示+追问 → 确认后写入

### 场景 7：版本发布
- **触发词**：发布、发版、打 tag、v0.X.Y
- **路由**：📦 版本发布 → 简化链
- **步骤**：
  1. 确认未发布变更（读 CHANGELOG.md [Unreleased]）
  2. 确定版本号（读版本号规则）
  3. 更新 CHANGELOG.md（[Unreleased] → [vX.Y.Z]）
  4. 更新项目版本号（代码中的 VERSION）
  5. git tag + git push --tags

## Gate 输出格式速查

| Gate 位置 | 输出格式 |
|-----------|-----------|
| Phase 0 完成 | ✅ Phase 0: [分类] \| [版本] \| 影响: [范围] → 确认后进入蓝图对齐。 |
| Phase 0.1 完成 | ✅ Phase 0.1: 蓝图对齐完成。[一句话关系]。→ 确认后进入任务拆解。 |

| Phase 5 完成 | ✅ Phase 5: 记忆归档完成。[待实施/完成]。 |

## Commit 格式速查

| 类型 | 格式 | 示例 |
|------|------|---------|
| Bug 修复 | `fix(scope):` | `fix(kb_query): answer() 修复 LLM 配置读取` |
| 新功能 | `feat(scope):` | `feat(ingest): 新增两阶段智能摄入` |
| 重构 | `refactor(scope):` | `refactor(config): 配置系统迁移到 pydantic-settings` |
| 文档 | `docs(scope):` | `docs(readme): 更新 README 架构图` |
| chore | `chore(scope):` | `chore(deps): 升级 qdrant-client 到 1.12.0` |

**回滚**：永远用 `git revert`（不用 `git reset --hard`）。

## 审查回路检查清单

说出「看看有没有遗漏」/「审查一下」/「迭代一次」后，AI 按此清单检查：

1. **符合蓝图吗？** → 当前产出和 BLUEPRINT.md 的「当前重心」一致吗？
2. **有遗漏吗？** → 有没有没覆盖的路径/场景/文件？
3. **有矛盾吗？** → 不同步骤之间有没有前提条件冲突？
4. **有假设需要确认吗？** → 「我以为是这样」但没问过你的——列出来
5. **输出问题清单** → 编号列表，每个问题一句话

## 跨项目影响判断

| 改动涉及 | 可能影响其他项目？ |
|-----------|-------------------|
| `.env` / `settings.py` / `config/` | ✅ 是（配置变化可能影响所有项目） |
| 单个页面/单个函数的功能代码 | ❌ 否 |
| `requirements.txt` / `package.json` | ✅ 是（依赖变化影响所有项目） |
| 项目专属文档（README / CHANGELOG） | ❌ 否 |

## QuickFix 条件速查

同时满足以下全部条件，才可走 QuickFix 短路径：

- [ ] 🐛 PATCH 级别（不改功能，只修表象）
- [ ] ≤ 1 个文件
- [ ] ≤ 5 行改动
- [ ] 无逻辑变更（纯配置 / 拼写 / 常量修正）

**如果不满足** → 走完整链（Phase 0.1 → Phase 1 → Phase 2 → ...）

## 版本号规则速查

| 类型 | 变化 | 示例 |
|------|------|---------|
| 仅 Bug 修复 | PATCH (0.4.x) | 0.4.3 → 0.4.4 |
| 新功能 | MINOR (0.x.0) | 0.4.4 → 0.5.0 |
| 破坏性变更 | MAJOR (1.0.0) | 0.5.0 → 1.0.0 |

> 详细示例见 SKILL.md 的「版本号规则」段落。

## Mermaid 语法校验

每次生成/修改 Mermaid 图时强制执行：

- 禁止中文引号（`"` `"` → 改用无引号或英文引号）
- 检查节点 ID 不重复
- 检查无死链接（指向不存在的节点）

## 蓝图反哺清单（Phase 4 后必检）

1. **重心状态** — 当前重心有进展吗？（缓解了 / 没变化 / 暴露了新问题）
2. **新认知** — 这次改动有没有让你对程序的理解发生变化？
3. **边界偏移** — 这次改动有没有超出蓝图写的"不做什么"的边界？
