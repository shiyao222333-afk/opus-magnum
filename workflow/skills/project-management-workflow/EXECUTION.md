# 执行规则（EXECUTION.md）

> 完整链专用。从 Phase 0 进入完整链后读此文件。

---

## Phase 0.1：蓝图对齐（仅完整链）

1. **判断当前项目** — 不确定就问
2. 读 BLUEPRINT.md，输出：「当前重心是 [X]。本次任务 [一致 / 偏离（必要维护）/ 偏离（需讨论）]」
3. **跨项目检查**：改动涉及 `.env` / `config/` / `requirements.txt` → 「⚠️ 可能影响其他项目，需要确认吗？」

---

## Phase 0.5：代码探索

改前 MUST 先 Read / Grep / Glob 代码，理解项目结构和现有实现。NEVER 未探索就动手。

---

## Phase 1：任务拆解

1. **前置步骤：研究执行方式** — 搜竞品 / 查参考书籍 / 验证是否适用于本项目
2. **第三方库 API 验证** — 方案中用到的第三方库 API → MUST 验证可用性（同 DEBUG.md 规则）
3. 输出任务清单（表格：# / 做什么 / 在流程图的哪 / [MVP必须] 或 [锦上添花]）
4. 通俗解释（一句生活比喻，说清这次改动的意义）
5. 列出模糊点（或"方案清晰，无需追问"）
6. 确认后进入 Phase 2

---

## Phase 2：执行

**执行纪律**：
1. **CRITICAL: 改前先读** — Read 后再 Edit，NEVER 猜代码
2. **CRITICAL: 重试 ≤ 3** — 同一步骤失败 3 次 → MUST 换方案或停下来询问用户
3. **每步自查** — 改完立即输出「✅ 步骤 N/M 完成 — [做了什么]」
4. **CRITICAL: 遇阻 NEVER 自行跳过或改方案**

**遇阻升级规则**：
```
## ⛔ 遇到阻塞
阻塞点：[具体描述]
影响：[对当前任务的影响]
选项：① 继续尝试  ② 跳过（需你同意）  ③ 换方案（需你同意）
```
用户不选 → AI NEVER 自作主张。

**[自审]**（每步完成后立即做，不单独占 Phase）：
```
[自审] 遗漏？调试残留？改动正确？→ ✅ 通过 / ❌ 发现问题
```

**沟通规则**（每步必须遵守）：
- NEVER 用技术术语（用生活比喻）
- 代码改完翻译回蓝图：「改了 [X]，实现了蓝图中 [Y]」
- 完成后说清：「你可以这样验证：[具体步骤]」

**多个步骤组合执行时**（如"A+B"），每步骤完成后用 `---` 分隔线输出独立的「步骤结果」区块。

---

## Phase 3：验证

```
## 验证结果

自动语法检查：✅ PASS / ❌ FAIL
你可以这样验证：[打开 X / 点击 Y / 输入 Z，看到什么说明好了]

L4 用户验收：请确认。
```

用户说 **"好了"** 才算通过。AI NEVER 自己说"验证通过"。

---

## Phase 4：文档更新

| 文件 | 什么时候更新 |
|------|-------------|
| `CHANGELOG.md` | 每次代码变更 |
| `PROJECT_PLAN.md` | 功能完成 / 版本发布 |
| `BLUEPRINT.md` | 蓝图反哺检查发现需要更新时 |

**技术债记录**：跳过的任务直接记入 PROJECT_PLAN.md 的「已知问题」段落。NEVER 创建独立的 TECH_DEBT.md 文件。每次 Phase 4 检查是否有技术债需要记录。

**Mermaid 语法校验**（每次修改 Mermaid 图时强制执行）：
- 禁止中文引号（`" "` → 改用无引号或英文引号）
- 检查节点 ID 不重复
- 检查无死链接（指向不存在的节点）

**蓝图反哺**（3 条检查清单）：
1. 重心状态 — 当前重心有进展吗？（缓解了 / 没变化 / 暴露了新问题）
2. 新认知 — 这次改动有没有让你对程序的理解发生变化？
3. 边界偏移 — 这次改动有没有超出蓝图写的"不做什么"的边界？

---

## Phase 5：记忆归档

写 **一句**到今天的 memory 日志：

```
- [做了什么] — 结果：[成功/阻塞/待实施]
```

如果还有后续任务 → 末尾加 **"待实施：[下一步]"**，方便明天恢复。

---

## QuickFix 短路径

同时满足以下全部条件，才可走 QuickFix：

- 🐛 PATCH 级别（不改功能，只修表象）
- ≤ 1 个文件
- ≤ 5 行改动
- 无逻辑变更（纯配置 / 拼写 / 常量修正）

```
Phase 0 分类 → QuickFix → Phase 2（执行 + 自审）→ Phase 3（验证）→ Phase 4（文档）→ Phase 5（归档）
```

---

## 中断恢复（对话中途纠正/追问/补充时触发）

收到纠正/追问 → 回到 Pre-flight（读蓝图 + 流程图 + 日志）→ 重新 Phase 0 分类 → 输出新分类 → 继续

---

## 长对话检查点

同一任务对话累计 15 轮后，MUST 输出进度摘要：

```
## 📋 长对话检查点（第 N 轮）
当前任务：[一句话]
已完成：[列表]
当前假设/方案：[最新状态]
已排除：[列表]
下一步：[一句话]
```

如果上下文已混乱（重复之前的错误判断）→ MUST 建议：「对话较长，建议开新对话，以上摘要作为起点。」

> 触发判断：输出 Phase 0 分类 15 次以上，或察觉上下文极长 → 主动输出检查点。
