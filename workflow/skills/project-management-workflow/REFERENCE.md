## 流程控制指令

| 用户说的话 | 作用 | 适用时机 |
|-----------|------|----------|
| 确认 / 继续 / ok / 好 | 推进到下一 Phase | 任何 Gate |
| 全部执行 / 一口气做完 | 跳过剩余所有 Gate，直到 Phase 5 | Phase 1 之后 |
| 修改 / 改一下 / 不对 | 回到 Phase 1 重新拆解 | Phase 1.5 之后 |
| 跳过 | 跳过当前 Phase（仅 Pre-flight / Phase 1.5） | Gate 前 |
| 先不推进 / 等等 | 停在当前位置 → AI 总结已有进展 → 等待新指令 | 任何 Gate |
| 回退 / 后退一步 | 回到上一个 Phase | 任何 Gate |

## 多轮多角度审查协议

### 触发条件

| 触发词 | 行为 |
|--------|------|
| 「看看有没有遗漏」 | 完整审查（自动选择新角度） |
| 「审查一下」 | 完整审查（自动选择新角度） |
| 「再审查一遍」 | 新一轮审查，MUST 使用新角度，NEVER 重复 |
| 「迭代一次」 | 轻量审查（仅 5 项快速清单） |
| 「汇报一下状态」 | 极简版：只说当前进度 |

### 核心规则

1. 每轮 MUST 换角度 — NEVER 两次从同一个角度审查
2. 角度由 AI 主动选择 — 选最能发现剩余问题的角度
3. 累计问题记入 PROJECT_PLAN.md — 所有发现的遗漏汇总到「已知问题」表

### 5 个标准审查角度

| # | 角度名称 | 审查什么 |
|---|---------|---------|
| **A1** | 清单对照 | 需求清单 / 验收标准 / 阶段目标 vs 代码实现 |
| **A2** | 链路追踪 | 从入口到出口逐行追踪数据流 |
| **A3** | 交叉校验 | schema vs 代码 / 文档 vs 代码 |
| **A4** | 防御纵深 | 边界条件 / 异常路径 / 极限输入 |
| **A5** | 架构边界 | 模块职责是否泄露 / 依赖方向是否正确 |

> 详细审查清单见 `SKILL-CHEATSHEET.md` 的「审查回路检查清单」段落。

## 主动提问（AI 自动触发，不等你问）

**检查点 1：Phase 0.1 蓝图对齐后**
- 蓝图有模糊点，或与当前任务不一致 → 追问

**检查点 2：Phase 1 任务拆解后、输出前**
- 扫描方案，列出所有"必须你拍板"的模糊点
- 全部清晰 → 输出：「方案清晰，无需追问。确认后开始执行。」

**提问格式**：
```
有以下几点需要确认：
1. [问题] — [为什么需要确认] — [AI 的建议方向]
```

## AI 沟通规则（每次输出必须遵守）

1. **NEVER 用技术术语** — 用生活比喻代替（如：不用"装饰器"，说"给函数加自动检测外壳，像给快递贴自动签收标签"）
2. **每步通俗解释** — 像给不懂代码的人解释
3. **代码改完翻译回蓝图** — 「改了 [X 文件名] 的 [Y 部分]，实现了蓝图中 [Z 目标]」
4. **优先用表格和编号列表** — NEVER 用大段文字
5. **方案清晰时直接执行** — NEVER 反问"要开始吗""确认吗"，直接动手
6. **git push 自动执行** — commit 后立即 push，NEVER 询问用户

## Git 提交规范

| 触发条件 | 操作 |
|-----------|------|
| Phase 3 L4 通过 + 提交前质量检查通过 | `git add -A && git commit && git push` |

**提交前质量检查**（5 项全通过才提交）：
- L1 语法 ✅
- L4 用户验收 ✅
- 代码自审 ✅（Phase 2 的 [自审] 全部通过）
- 无调试残留 ✅
- 文档已更新 ✅

**Commit 格式**：`type(scope): description`（Conventional Commits）
- `fix(` — Bug 修复
- `feat(` — 新功能
- `refactor(` — 重构
- `docs(` — 文档
- `chore(` — 杂项

**回滚**：NEVER 用 `git reset --hard`，MUST 用 `git revert`。
