# 调试协议（DEBUG.md）

> 调试模式专用。从 Phase 0 进入调试模式后读此文件。

---

## 调试模式概述

调试模式跳过蓝图对齐和任务拆解（bug 不需要对齐蓝图），但 **MUST 走完整证据链**。

---

## 第 1 步：Thinking Phase（行动前思考）

第一次 Read 代码前，MUST 输出思考块：

```
## 调试思考
现象：[用户报告的症状]
可能根因（列出 2-3 个假设）：
  1. [假设 A]
  2. [假设 B]
  3. [假设 C]
优先验证：[先验证哪个，为什么]
```

---

## 第 2 步：证据链（改代码前 MUST 输出）

```
## 证据链
现象：[用户报告的症状]
假设：[我认为根因是 X]
验证方法：[我怎么证明假设对不对——Read 代码 / 写测试 / 查文档]
验证结果：[证据支持 / 推翻了假设]
结论：[根因确认 = X / 假设排除，换下一个]
```

NEVER 没有证据链就动手改代码。

---

## 第 3 步：假设追踪表（第 2 次尝试起 MUST 输出）

同一 bug 修复第 2 次尝试时，MUST 输出假设表：

```
## 假设追踪
| # | 假设 | 验证方法 | 验证结果 | 状态 |
|---|------|---------|---------|------|
| 1 | [假设 A] | [怎么验证] | [结果] | ❌ 排除 |
| 2 | [假设 B] | [怎么验证] | [结果] | ⚠️ 部分正确 |
| 3 | [假设 C] | [怎么验证] | [结果] | ... |
```

**CRITICAL: 假设表填到第 3 行还没解决 → MUST 停下来**，向用户报告：
```
## ⛔ 已尝试 3 个假设均未解决
建议：① 切换模型  ② 重新从头研究  ③ 换技术方案
你选哪个？
```

NEVER 在第 3 行后继续猜。

---

## 第三方库 API 验证（调试时高频触发）

用了第三方库的 API（非 Python 标准库）→ MUST 先验证：
- 该 API 在当前版本**存在**
- 该 API 的参数 / 返回值**符合预期**
- 该 API **不依赖额外插件 / 配置**

验证方式：查官方文档 或 写一行测试代码跑。

> 典型踩坑：`dayjs().startOf("isoWeek")` 未加载插件静默返回当前时刻不报错。

---

## 调试 mini-loop

每步改完后 MUST 回答：「这个结果是否改变了我的根因假设？」
- 是 → 更新假设表，重新走证据链
- 否 → 继续下一步

---

## 调试注入器（每 5 轮提醒）

调试模式下，每次输出「调试思考」或「证据链」后，在末尾标注`（调试轮次：第 X 轮）`。

每累计 5 轮 → MUST 注入流程检查提醒：

```
⚠️ 调试检查点（第 5N 轮）
- 当前是否在按 DEBUG.md 走？
- 证据链是否每次都输出了？
- 假设追踪表是否及时更新了？
- 有没有偷偷跳过流程直接改代码？
```

> 目的：对抗「对话一长就忘记流程」。每 5 轮强制把流程规则拉回注意力。

---

## 前端任务验证规则

前端改动（JS / CSS / HTML / 第三方 UI 库）额外要求：

1. **改完 MUST 说明：预期效果 vs 实际效果** — 如「预期：列头显示"周一 7/7"，实际：只显示"周一"」
2. **涉及第三方 UI 库 → MUST 查文档确认 API** — 比后端更严格，AI 对前端渲染机制理解系统性弱于后端
3. **涉及时区 / 日期 → MUST 写测试验证** — 典型坑：UTC 存储 + 本地显示、`toDate()` 跨天、`startOf()` 插件未加载

> 典型踩坑：DayPilot `weekStarts` 默认 0（周日），`locale: "zh-cn"` 不覆盖；`dayjs().startOf("isoWeek")` 未加载插件静默返回当前时刻。
