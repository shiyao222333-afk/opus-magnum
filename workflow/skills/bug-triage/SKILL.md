# 疑难杂症处理流程 (bug-triage)

> 通用救援子协议。当同一个 Bug 多轮修改仍未修复时启用，嵌在项目管理流程 Phase 2 执行卡住时触发。
> 不替代项目管理流程，而是把"重试≤3 换方案" operationalize 成带记录文件的强制流程。

---

## 何时启用

满足任一即开：

- **显式信号**：你说「记录失败的修复 / 又失败了 / 把所有失败经验记下来」→ 立即开
- **隐式信号**：同一症状连续 ≥3 次修改都未通过验收 → 自动开

一次性 Bug、QuickFix 范畴的不开。

---

## 流程骨架

```
[触发] 显式(你说"记录失败") 或 隐式(同症状≥3轮未过验收)
   ↓
[开流程] 先读已有 BUGLOG + 已排除清单   ← 防重复踩坑，不是立刻改代码
   ↓
[开临时记录] 建 <项目>/.workbuddy/memory/BUGLOG-<主题>-<MMDD>.md
             置顶「已排除清单」+「假设银行」
   ↓
┌─ 循环：每轮
│   ├─ 假设银行选 1 个未验证根因      （一次只验一个）
│   ├─ 只改 1 个变量验证              （单变量隔离，结果可归因）
│   ├─ 附诊断快照（工具形式不限，输出必须一键复制纯文本）
│   ├─ 按模板追加：假设/改动/诊断/结果/根因/状态
│   ├─ 已证伪 → 写入「已排除清单」(永不重试)
│   └─ 同方向≥3败 → 强制换思路
└─ 直至修复成功（诊断达标 + 你验收）
   ↓
[收敛] 标记关闭 → 精简(留症状/根因/方案/教训)
        → 双写归档（见下方「归档位置」）
   ↓
归档即终点 — 不再修改本 skill 文件（见下方「SKILL 文件保持固定」）
```

---

## 三条铁律

1. **开流程先读记录** — 下一步是读已有 BUGLOG + 已排除清单，不是改代码。绝不复试已证伪方向。
2. **单变量隔离** — 每轮只改一处，结果才能归因到具体改动。
3. **假设银行** — 候选根因列清单，标 未验证/验证中/已确认/已排除，每轮只验一个。

---

## 诊断工具原则（广义）

- 任何疑难 Bug 都**应有**一个诊断工具（形式不限：UI 面板 / CLI 命令 / 脚本 / 接口返回 / 日志转结构化）。
- 诊断输出**必须能一键复制为纯文本**，直接贴回对话。禁止依赖截图、禁止让人手敲数字。
- 为什么：纯文本可 grep、可跨会话检索、不易错；截图和手敲做不到。

---

## 归档位置（双写）

归档文件（`BUGLOG-<主题>-<MMDD>-FIXED.md`）必须同时存在两处：

1. **项目文件夹**：`<项目>/.workbuddy/memory/BUGLOG-<主题>-<MMDD>-FIXED.md`
   （跟随项目走，PM 流程 Phase 0.5 重开任务时自动读到）
2. **巨作统一管理目录**：`D:/opus-magnum/docs/bug-triage-cases/BUGLOG-<主题>-<MMDD>-FIXED.md`
   （跨项目集中管理，作为疑难 Bug 案例库，便于横向检索同类坑）

两处内容一致，任一即可作为证据。新建案例前先确认巨作目录存在（没有就 `mkdir`）。

---

## SKILL 文件保持固定

- 本 skill（SKILL.md / DEBUG-LOG-TEMPLATE.md / ANTI-PATTERNS.md）是**方法论、相对固定**。
- **收敛步骤不再包含「提练教训进 skill 踩坑库」**——那是错的：每次 bug 修复都去改 skill 文件，会让 skill 不稳定、无限膨胀。
- **归档文件（BUGLOG-FIXED）本身就是沉淀载体**，足够。教训写在归档文件里即可，不要再回写 skill。
- 只有在「新建 skill」或「大修本流程」时才动 skill 文件，不是每次 bug 收敛都动。
