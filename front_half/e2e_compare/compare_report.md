## 二、三次输出对比：到底差在哪

### ① 馏析（Nigredo）—— 完全一致 ✅
- run1 与 run2 的 transit① **md5 相同**（`10523f68…`），字幕逐字一致。
- 原因：同地址二次命中视频缓存，直接复用已转写字幕，不重跑 Whisper。
- **结论：馏析层确定性强，重复处理幂等、零冗余。**

### ② 炼真（Albedo）—— 裁决翻转 ⚠️ 重大发现
同一份 transit①（一字不差），两次跑出**截然相反的真相裁决**：

| 字段 | run1 | run2 | 差异 |
|------|------|------|------|
| `status` | **rejected**（不可信/虚假） | **accepted**（可信/真实） | ❌ 翻转 |
| `truth.label` | false | true | ❌ 翻转 |
| `truth.score` | 40 | 85 | 差 45 分 |
| `truth.grade` | L1（仅作者声称） | L3（多源/权威） | 跳两级 |
| 摘要要点数 | 7 | 7 | 一致 |
| 优点分析维度 | 8 项 | 8 项 | 一致 |
| `structure_type` | sop | sop | 一致 |
| 变现检测 | 未检出 | 未检出 | 一致 |

- **结论：炼真非确定。** 内容、结构、优点维度都一致，唯独「真实性裁决」从「假」翻到「真」。
- **根因**：真实性评估是 DeepSeek LLM 调用，未固定随机种子（temperature>0），同输入不同次结果漂移。
- **危害**：炼真是「质检关卡」，同一条视频今天判假、明天判真，把关完全不可信。这是比熔知 hook bug 更根本的可靠性缺陷。

### ③ 熔知（Citrinitas）—— 幂等去重 ✅（但暴露交互问题）
- run1：写入 **9 个分块点**，`epistemic_status=rejected`、`trust_score=0.0`（已用修复后 hook 更正，见上轮）。
- run2：检测 doc_id 重复（doc_id 由文件名决定，两次同名→同 id），**直接跳过，KB 不变**。
- **结论：熔知层幂等去重正确**，同文件重复摄入不产生重复点。
- **暴露的交互问题**：因为 doc_id 按文件名定（非内容），而炼真裁决又漂移——**KB 会锁定「第一次跑」的裁决**。run2 的新裁决（accepted）没能覆盖 run1（rejected）。若要「修订后重摄入生效」，去重键需改为内容哈希，或摄入前先删旧点。
